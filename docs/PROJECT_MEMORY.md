# Candidate B fix — LANDED on branch new-video-on-fix-egret (2026-06-02)

Implements the [[overlay-stuck-root-cause]] successor: the phantom-$0-bank self-clobber
(see prior session). Root cause: our `addrController/addrDecoder` mapped motherboard RAM at
`$0` (`in_motherboard_low`, commit 2206dfb) as a **live writable mirror of `$800000` from
reset**, so the boot ROM's bank probe found a phantom 2 MB bank at `$0`, prepended
`{$0,$200000}` to the descriptor table (ptr `$9FFFE4`), and marching it clobbered the real
`$800000` table at `$9FFFEC` via the `$0↔$800000` SDRAM mirror → garbage 2nd entry → hang
at `$A46862`.

MAME ground truth (`mame maclc -ramsize 2M`, taps in /tmp/tap*.lua):
at RAM-enumeration time MAME's `$0` is EMPTY (`ram_size(0xc0)` on overlay-clear installs
only `$800000-$9FFFFF` for 2 MB); the `$0` mirror appears only when the ROM writes the V8
config register (`via2_config_w` → `ram_size(data)`). MAME table = single entry
`[$800000,$200000,$FFFFFFFF]` @ `$9FFFEC`, SP=`$807FFC`, march A0=`$800000`/A1=`$9FFFEC`.

## The fix
`ram_configured` latch in `rtl/pseudovia.sv` (reset 0; set 1 on ROM write to native config
reg `$01`/ram_cfg — single always block, Quartus-safe). Gates **only** `in_motherboard_low`
in `rtl/addrDecoder.v` (NOT `in_simm_range` — 10 MB config has a real SIMM at `$0`). Threaded
via `rtl/addrController_top.v` and `MacLC.sv` (`pvia_ram_configured`). 4 files, ~25 lines.

VERIFIED (clean build, `--screenshot 750 --stop-at-frame 800`): table single-entry @ `$9FFFEC`
(phantom `$0` entry GONE), SP=`$807FFC`, march `$800000→$9FFFEC`, `*** DONE` x2 (RAM + VRAM
passes, like MAME's multi-pass), no `$B6DB6DB6`, frame-750 no longer black (orange now).
All orchestrator checklist items pass.

## NEW downstream blocker (next session) — NOT the RAM bug
Boot now advances past `$A46862` but stalls at **`$A499E8`** (`btst #16,d7; beqw $A49A6E →
braw $A498EC` poll loop, waiting for D7 bit16 that never sets; 36 identical PC samples
F557→F800). Path: march → `$A47084` → `$A4A444` → `$A471C2` → `$A469xx` → `$A15432` →
`$A499E8`. `$A15424` just before is a peripheral handshake (`btst #3,(A1); beq self`; dbf

branches into a path MAME doesn't, likely a peripheral-status (Egret/VIA/ADB) divergence, not

## FILE: egret_protocol_fix.md

---
name: Egret Protocol Fix
description: Egret uses xcvrSes/viaFull/sysSes protocol (not CUDA TREQ/TACK/TIP); response needs type+flags+cmd minimum 3 bytes; Apple EgretMgr.a is the authoritative source
type: project
---

The behavioral Egret boot hang was caused by two issues, resolved 2026-03-30:

1. **Wrong protocol**: Was using CUDA protocol (TREQ/TACK/TIP + SEND_NOTIFY dummy bytes). Real Egret uses xcvrSes (PB3)/viaFull (PB4)/sysSes (PB5) with viaFull rising/falling edge handshaking. No dummy bytes needed — xcvrSes deassert signals end-of-transaction.

2. **Wrong response format**: Was sending [PKT_ERROR, data] (2 bytes). Host's `SendEgretCmd` reads exactly attn+3 bytes (type, flags, cmd). Correct format: [type_echo, flags, cmd_echo, data...] minimum 3 bytes. Type must echo original ($01 for pseudo) not $02 (error), or host aborts init.

**Why:** Apple's `EgretMgr.a` (SuperMario source) is the definitive reference for the host-side protocol. The equates in `egretequ.a` define signal names: xcvrSes=PB3, viaFull=PB4, sysSes=PB5.

**How to apply:** If Egret communication breaks again, consult `<HOME>/repos/supermario/base/SuperMarioProj.1994-02-09/OS/EgretMgr.a` and `Internal/Asm/egretequ.a` for the authoritative protocol.

# Egret CB1 Debug Investigation

## Problem
ROM hangs at PC=$A14E1A polling VIA IFR bit 2 (SR interrupt). The VIA shift register
is in mode 3 (shift in, external clock CB1) but CB1 never rises.

## Evidence from logs (newlogsdedup.txt)
1. Egret HC05 CPU is running (HC05 TRACE messages show PC advancing)
2. All `EGRET[n]: PB OUT` messages show `CB1=0` — never goes to 1
3. VIA SR is triggered: `VIA: SR triggered - mode=3, dir=0, ext_clk=1`
4. SR shift_active=1, bit_cnt=7 — waiting for first CB1 edge
5. VIA IFR reads return $02 (bit 1 set = slot/timer, bit 2 = SR = NOT set)
6. TREQ toggles show Egret is alive and communicating via Port B

## ROM code at hang point
```
A14E1A: btst  #2, ($1A00,A1)   ; test IFR bit 2 (SR interrupt)
A14E20: beq   $A14E1A          ; loop if not set
A14E22: bset  #0, (A1)         ; continue when SR fires
```
$F01A00 = VIA register 13 = IFR

## Sequence before hang
1. ROM writes ACR=$1C (mode 7: shift out, ext clk) — initial setup
2. ROM writes ACR=$0C (mode 3: shift in, ext clk) — switch to receive
3. ROM reads SR (triggers shift_active=1)
4. Polls IFR bit 2 forever — CB1 never clocks

## Key wiring (dataController_top.sv)
- `wire via_cb1_in = cuda_cb1;` — VIA CB1 input comes directly from CUDA/Egret
- Egret drives CB1 via Port B output
- Two implementations: `egret_wrapper` (ifdef USE_EGRET_CPU) or `cuda_maclc` (default)
- Verilator sim uses `cuda_maclc` (state machine, not real HC05)

Wait — the logs show `HC05: TRACE` messages, suggesting USE_EGRET_CPU IS defined for sim.
Need to check which implementation is actually compiled.

## Next steps
1. Determine which Egret impl is active (USE_EGRET_CPU or cuda_maclc)
2. Find why CB1 is never toggled — check Egret Port B DDR and output register mapping
3. Check if CB1 bit position in Egret's Port B matches what the Egret firmware expects
4. Look at egret_wrapper.sv / cuda_maclc.sv for how PB maps to cuda_cb1

# STM boot failure ROOT CAUSE (2026-06-02) — MOVES CPU-space probe needs BERR

Supersedes the "STM is just the reporter / find the failing self-test" framing in
[[../../../../repos/MacLC_MiSTer/docs/resume_stm_060226]]. The failing test is a **bus-error
hardware probe**.

## The mechanism (proven via MAME vs our cpu_trace.log)
Boot ROM machine-ID routine (runs ~F407 in sim) executes a hardware probe:
```
A03A82: move.b #$7,D1
A03A86: movec  D1,SFC          ; Source Function Code = 7 (CPU space)
A03A8A: moves.w ($00022000).l,D1 ; read $22000 in CPU space  -> MUST BUS ERROR
...                              ; BERR handler routes to the test-result dispatch
A46240: btst #$1b,D7; beq $a4624a; movea.l A5,A7; jmp (A6)  ; A6=$A03AB4 (pass→recover)
A03AB4: beq $a03aba
A03AB6: bclr #$1c,D2            ; ROM bytes 0882 001C — clears BIT 28 of config word
```
- **MAME**: `moves.w $22000` ($A03A8A) bus-errors **13/13 times** → always lands on `$A46240`.
- **OURS**: same probe runs 22×, **always falls through inline to `$A03A92`** (cmp.b D1,D1; jmp(A6));
  `$A46240` executes **0 times**. Our moves logs effective data addr `$55FE000B` (garbage).

## Consequence chain
Config-word template in ROM at `$A03BB4` = `$DC000D07` (loaded `move.l ($10,A1),D2`).
- MAME (probe BERRs correctly) → `$CC000D07` (bit 28 cleared, low byte stays `07`).
- Ours (no BERR) → `$DC000D1F` (bit 28 left set + low byte corrupted `07`→`1F`).
D2 low byte is a dispatch index into an **8-entry** jump table at ROM `$A00AB4`
(`move.b D2,D5; lsl #1; move.w (6,PC,D5),D5; jmp (2,PC,D5)`). Index 7 → `$A00AC4` (continue=MAME).
Our index `$1F`(31) reads past the table → word `$3120` → jmps into descriptor data `$A03BD4` →
runs garbage → error entry `$A46280` (code `$0A`) → `$A4638C` → `$A48CDA` fatal → STM monitor
(`*APPLE*` over SCC). At the dispatch ($A00AB0) EVERY register matches MAME except D2
(D0=773F D1=1A6 D4=FFFFFFFF A0=A03AE4 A1=A03BA4 all identical).

## The fix — PART 1 DONE, PART 2 is the real blocker
**Part 1 (bus glue, DONE in both files):** assert BERR for FC=7 CPU-space accesses that aren't
IACK. Added `wire fc7_iack = (cpuFC==3'b111) && (cpuAddr[19:16]==4'hF);`, made `_cpuVPA` assert
only for `fc7_iack`, made `cpu_berr = (cpuFC==3'b111) && !fc7_iack && !_cpuAS`. Applied to BOTH
`MacLC.sv` (FPGA) AND `verilator/sim.v` (sim — see [[verilator-top-is-sim-v]]). **`sim.v` also
had `.berr( 1'b0 )` hardwired — changed to `.berr(cpu_berr)`.** Correct + necessary, no regression.

**Part 2 (the actual blocker — NOT yet fixed): TG68K MOVES is broken for absolute-long EA.**
Verified by probing the kernel (`rtl/tg68k/tg68k.v` wrapper + sim.v): for the `moves.w $22000`
probe the kernel **never presents FC=7** and **never accesses $22000** — the trace shows the
moves reading a garbage EA (`$55FE000B`) and completing inline. So Part-1's BERR can't fire
(FC stays normal=5). MOVES decode (`TG68KdotC_Kernel.vhd:1779-1797`) does `getbrief` (fetch the
ext word) AND `set(ea_build)` in the SAME cycle, then schedules `next_micro_state<=moves1`
(which sets `set_exec(moves_fc)` → FC=SFC). HYPOTHESIS: for a **multi-word EA (absolute long)**
the ext-word fetch and the EA address-word fetch collide / `ea_build` overrides `next_micro_state`,
so `moves1`/`moves2` never run → FC override never applied → wrong EA + FC=normal. Register-mode
MOVES (the other ROM `moves` at A03C4E etc., (d16,An)/(An)+) likely work; only abs-long breaks.
`moves_fc`=exec slot 89, fits the 90-bit `exec` vector (`lastOpcBit=89`), so NOT an overflow bug.

**FIX SOURCE FOUND (2026-06-03):** `../Minimig-AGA_MiSTer/rtl/tg68k/TG68KdotC_Kernel.vhd` is a
fully-debugged descendant of this same kernel with explicit fixes for our exact bugs — **BUG #322**
(MOVES complex/abs EA overwritten before moves1 → `moves_ea_latched`), **BUG #149** (FC-override
window → `moves_bus_pending`/`moves_fc_override`), **#318** (latched ext-word fields), **#214**
(writeback guard). Not a drop-in (10k vs 4k lines, adds FPU/PMMU/cache, different entity ports);
**port the MOVES + movec-SFC/DFC logic into ours.** Confirmed-broken in ours via encoding-independent
probes: `moves_fc_active` (exec 89) never asserts AND `SFC` never leaves its $5 default → whole
68010+ MOVES/MOVEC path is a half-finished port. Regen toolchain validated (ghdl 6.0.0 deterministic;
micro_state enum = declaration order, moves1=91/moves2=92). Full port plan in
docs/resume_stm_060326.md.

**Proper fix (superseded by the Minimig port above):** in the kernel microcode, fetch the MOVES extension word FIRST, THEN build the EA,
THEN run moves1/moves2 — so FC=SFC is presented during the data cycle. Edit VHDL (source of truth)

- `A467F2 cmp.l (A0),D0`    — reads `$50F40000` = `$5368656C` → SHOULD equal D0=`$5368656C`

doesn't flush/advance the compare operand between two consecutive longword reads to the same base
register. `last_data_read` is a red herring (it tracks opcode fetches, not the data operand).

Commit A (`4ae3f06`) landed cpuAddr widening + diagnostic infra, but the headline value (real BERR-on-unmapped) is **disabled**. Engaged plan's escape hatch.

**Why:** Enabling `(!_cpuAS && cpuBusControl && selectUnmapped)` in `cpu_berr` regresses boot to uniform-grey screen. CPU storms BERR on these 4 addresses during early ROM init:

| Address    | FC  | RW | Notes |
|------------|-----|----|----|
| `$000000`  | 101 | 1  | supervisor data read; overlay-timing artifact, ROM is supposed to be there |
| `$000002`  | 101 | 1  | same |
| `$F21C00`  | 101 | 1  | unmapped peripheral hole between SWIM ($F16xxx) and Ariel ($F24xxx). Likely a real LC peripheral we don't model |
| `$FC0000`  | 101 | 1  | gap between VRAM end ($FBFFFF) and $Fxxxxx peripheral region |

Diagnostic captured 41 events during boot; the unique-address set above is what BERR would trip on.

**Why BERR can't just be re-enabled:**

1. **TG68 68020 mode is incomplete.** `rtl/tg68k/TG68KdotC_Kernel.v` line 218 comment: `// 00->68000  01->68010  11->68020(only some parts - yet)`. 68020 BERR needs format $A/$B stack frames (~46–92 bytes with pipeline state). TG68 likely emits a 68000-style 14-byte frame → ROM's BERR handler reads garbage off the stack → CPU explodes.

2. **`clr_berr` is dangling** in the wrapper. `rtl/tg68k/TG68K.v:217` — `.clr_berr ( /*tg68_clr_berr*/ )`. The kernel's BERR-latch ack path is unwired. Not the immediate blocker (our `cpu_berr` is combinational and drops with AS), but means there's no host-visible "BERR consumed" handshake.

3. **`IPL_autovector` is hard-tied to `1'b0`** at `TG68K.v:215`. Autovector is implemented via the BERR-during-IACK convention (`cpuFC == 3'b111`) rather than the proper input. That's why flipping `sim.v`'s `.berr(1'b0)` to autovector (which would have given parity with `MacLC.sv`) regressed sim boot — IRQ handling depends on the existing path.

**Possible peripherals we don't model that might own these addresses:**
- `$F21C00` is in the V8 register region. Could be a V8 internal register, VIA2 mirror, or PDS-related. Worth cross-referencing `schematics/LC_Master_Netlist.csv` and the V8 schematic audit notes.
- `$FC0000` could be a mirror of something earlier in $Fxxxxx, or sense input space. Less likely a missing peripheral, more likely a probe.
- `$000000`/`$000002` is overlay ROM territory — these might fire post-overlay-disable from a stale fetch, in which case it's a timing bug not a missing device.

**Paths forward (in order of cost):**

(a) **Sidestep — pre-filter at decoder.** Make `addrController_top`/`addrDecoder` return a quiet dummy read (e.g. `$FFFF`) for the 4 known addresses instead of `selectUnmapped`. Cheap, unblocks FPGA `0xCxxxxx` debug. Doesn't fix the underlying TG68 BERR issue.

(b) **Investigate $F21C00 specifically.** Check schematics/MAME `v8.cpp` decode table for what real hardware does with that address. Might be a known register we can stub.

(c) **Audit TG68 68020 BERR + wire `clr_berr`.** Read `TG68KdotC_Kernel.v` (GHDL-converted Verilog from `how-to-convert-cpu.txt`) for `trap_berr`, stack frame format generation. Hard.

(d) **CPU core swap.** No mature open synthesizable 68020 core known. Musashi is software-only. This is a multi-week effort and is the wrong tool for unblocking the boot.

**How to apply:** When resuming Commit A or chasing the FPGA `0xCxxxxx` stall (see `fpga-boot-debug` memory), start with (a) to get visibility, then (b) to understand $F21C00. Don't re-enable real BERR until at least the TG68 stack frame question is answered.

- `fpga_boot_debug_2026_03_21.md` — original $0xCxxxxx stall that motivated BERR work
- `4ae3f06` — what actually landed

# Verilator builds `sim.v`, not `MacLC.sv`

The Verilator simulator compiles `--top-module emu` from **`verilator/sim.v`**, and the source
list in `verilator/Makefile` does **NOT include `MacLC.sv`**. `sim.v` defines its own `emu`
top-level harness with its **own copy** of the CPU instantiation and bus-termination logic
(`_cpuVPA`, `_cpuDTACK`, `cpu_berr`, the `tg68k` instance).

**Consequence:** an RTL fix to the CPU/bus glue in `MacLC.sv` is NOT exercised by the sim. Any
such change must be made in **both** `MacLC.sv` (FPGA/Quartus) and `verilator/sim.v` (sim), and
they can silently drift. Always check both when touching: VPA/DTACK/BERR, FC decode, cpuAddr
wiring, the tg68k port map, overlay/RAM-config threading.

Notable gotcha found 2026-06-02: `sim.v` had `.berr( 1'b0 )` hardwired on the tg68k instance —
the computed `cpu_berr` was never connected, so bus errors could never fire in sim (MacLC.sv
wired `.berr(cpu_berr)`). This masked the [[stm-root-cause-moves-berr]] MOVES-probe fix until
`sim.v` was also patched (IACK discriminator on `_cpuVPA` + real `.berr(cpu_berr)`).

Other sim-only files in the Makefile list: `sim.v`, `sim_ram.v`. RTL under `rtl/` (tg68k,
addrController_top, addrDecoder, dataController_top, pseudovia, via6522, etc.) IS shared.

also clk16-enable sampling + the TG68 wrapper's internal `tg68k.tg68_din_r` and `s_state`.
The probe is `cmp.l (A0),D0` at $A467F2, A0=$50F40000, D0=$5368656C (set at $A467CC, never
modified). It's a longword read = two 16-bit cycles: $F40000->SDRAM word $580000 (=$5368),
$F40002->$580001 (=$656C).
1. BASELINE (VPA path, no fix): dataControllerDataOut settles to the correct word at the
   first cpu-slot memoryLatch and is then ROCK-STABLE for 40+ clks (cpu_data only latches in
   cpu slots, so video fetches don't corrupt it). DTACK never asserts — VPA E-clock samples
   din at a fixed phase that catches the STALE prior word (e.g. $6708) before it settles.
2. Naive sim_ram has a 1-CLOCK REGISTERED-READ LAG (`dout<=mem[addr]`): when memoryLatch
   samples the cycle right after memoryAddr switches, ram_do is still the previous address's
   data. Fixed-DTACK-timing tweaks just MOVE the failure between the two longword words
   because changing DTACK re-aligns ALL subsequent bus phases.
3. Discovered the longword WRITE was ALSO truncated: with VRAM on DTACK, `mem[580001]` got
   written 0 times (only $580000 committed) — DTACK acked before the 2nd word's cpu write
   slot, so $656C was LOST and the readback legitimately returned 0.
4. WORKING SUB-FIXES (verified in CLEAN build): (a) combinational sim_ram read
   (`always @(*) dout=mem[addr]`) removes the lag; (b) give each VRAM access TWO cpu slots
   via a `vram_slot` counter (1st slot lets read settle / write commit, 2nd asserts DTACK).
   After these: `mem[580001]` IS written ($656C), and the dump shows dataControllerDataOut
   AND the TG68's own input register `tg68_din_r` latch the CORRECT values at state 6/7:
   din_r=$5368 for word1, din_r=$656C for word2. So the CPU's data-input register holds
   exactly $5368656C = D0.
5. **PARADOX (unresolved):** despite tg68_din_r = $5368/$656C (correct) for both words, the
   `cmp.l` STILL reports unequal — `$A467F4 beq` not taken, `$A467F6` executes, $4E754F00
   march persists. The data is correct at EVERY observable point incl. the TG68 wrapper's
   data register. So the remaining discrepancy is INSIDE the TG68 kernel
   (rtl/tg68k/TG68KdotC_Kernel.v, generated VHDL->Verilog) — how/when it consumes data_in to
   assemble the 32-bit operand and compare. Possibly a 68020 long-read assembly/timing issue,
   or D0 itself differs at the ALU. Black-box probing is exhausted.
