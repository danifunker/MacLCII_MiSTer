# Plan: V8 video overhaul — real 4/8bpp color, white-bar fix, true 640x480 (2026-06-05)

Symptoms being fixed: no color on HW; "VRAM corruption" at any depth above 2bpp;
intermittent white bar (left-edge artifact); 640x480 mode runs at ~38.7Hz, not
60Hz (the "640x480 issue" — scaler/RT4K unhappiness, shimmer).

## Root causes (all confirmed in RTL / by the numbers)

1. **Video fetch bandwidth ceiling.** The bus arbiter (`addrController_top.v`
   `busCycle`) grants video 1 of 4 SDRAM slots = one 16-bit word per 16 clk_sys
   = 2.03 Mwords/s @ 32.5 MHz. At the 16.25 MHz pixel-enable that is *exactly*
   2 bpp's appetite (`maclc_v8_video.sv:193` admits "That cadence only matches
   2 bpp"). 4/8 bpp are currently faked by stretching pixels (2:2:2:2 / 4:4
   shift splits) and 1 bpp by a dedup hack that renders 8 of 16 pixels.
   Per-line budget (640x480, line = 1600 clk_sys = 400 total word slots):

   | depth | words/line needed | share of total bus |
   |-------|------------------:|--------------------:|
   | 1bpp  | 40                | 10% |
   | 2bpp  | 80                | 20% (= today's full video share) |
   | 4bpp  | 160               | 40% |
   | 8bpp  | 320               | 80% |
   | 16bpp@512x384 | 512/1280clk | >100% — impossible without SDRAM burst |

   Real SDRAM (tRCD+CL, vs sim_ram's 1-cycle) eats the margin, which is why HW
   is worse than sim at the same depth.

2. **White bar / left-edge artifacts.** Pixels rendered before the first VRAM
   word of a line arrives come from a cleared/stale shift register whose
   *palette index* isn't guaranteed black (`maclc_v8_video.sv:233` — the 0xFFFF
   fill attempt caused exactly an "intermittent left-edge line on hardware").

3. **640x480 refresh.** Pixel clock = clk_sys/2 = 16.25 MHz → 800x525 frame =
   38.7 Hz. True VGA needs a 25.175 MHz dot clock (PLL has a spare tap).

4. **(found while planning) pseudovia video-config READ diverges from MAME.**
   MAME `v8.cpp:339`: read returns ONLY `montype << 3` (monitor sense in bits
   5:3, all other bits 0). Ours returns `(video_config & $C7) | (sense<<3)` —
   stored depth bits leak into the readback. If the OS/Monitors panel masks
   wrongly this can misreport the display and restrict the offered depths
   ("no color" in the Monitors panel). One-line fix; verify against MAME.

## Ground truths to extract from MAME first (Phase 0)

- `v8.cpp screen_update`: per-depth pen indexing (1bpp `pens[0x7F/0xFF]`-class
  mapping, 2/4/8bpp pen index construction), fixed 1024-byte stride for modes
  0-3, `hres*2` for 16bpp. We already match the stride; verify pen mapping per
  depth against the Ariel CLUT we feed.
- Depth legality matrix per monitor (what the Monitors panel offers for
  montype 6 = 640x480 VGA and montype 2 = 512x384) and the montype encoding
  used in the sense readback.
- Reference framebuffers: boot the SCSI 6.0.8 image in MAME, set Monitors to
  each depth (B&W, 4, 16, 256 colors), save (depth persists on disk/PRAM),
  snapshot each. Keep one prepared .hda **per depth** — the same disk then
  makes our sim boot straight into that depth (no UI interaction needed in
  Verilator). These are the oracle images for every later step.

## Phase 1 — scanline FIFO + request-driven video arbitration
*(fixes: >2bpp corruption, white bar, 1bpp dedup hack; shared RTL → sim and
core get it identically)*

1. **Line FIFO in `maclc_v8_video.sv`**: dual-port BRAM, 512x16 (one full
   1024-byte stride line). Fetch side fills ahead of the beam (start filling
   line N+1's words as soon as line N display begins; hblank+active time both
   usable). Display side pops at pix_en exactly `bits_per_pixel` bits per
   pixel — the shift-distribution hacks and the 1bpp dedup tracker are
   **deleted**, every depth gets its true horizontal resolution.
2. **Request-driven slots in `addrController_top.v`**: video asserts
   `video_req` while the FIFO is below its fill target. Arbiter policy:
   - slot 00: video (as today, unconditional priority),
   - slots 10 and 11: video **iff** `video_req`, else their current owners
     (dio, cpu),
   - slot 01: always CPU (hard floor of 25% so the Egret/VIA handshakes and
     the CPU never fully stall — bus-arb is the fragile zone, CLAUDE.md).
   ≤2bpp never asserts video_req beyond slot 00 → **bit-identical behavior to
   today at 1/2bpp** (regression containment). 4bpp uses ~2 slots, 8bpp up to
   3 (CPU slows during active video — the real LC slowed at 8bpp too; honest).
3. **White-bar fix**: display side forces RGB=0 (not a palette index) until
   the FIFO holds the line's first word and outside active video. Kills the
   left-edge artifact class palette-independently. Also revisit the ~5px
   left-edge artifact noted in [[desktop-stage-open-issues]].
4. **pseudovia readback**: align video-config read to MAME (`sense<<3` only).
5. 16bpp: explicitly out of scope (needs Phase 3 burst); gate it off cleanly
   (treat mode 4 as 8bpp or blank with a sim warning) so it can't wedge boot.

**Validation (every step, sim first):**
- `check_boot.sh --run 100` + frame-350 screenshot (Egret/VIA regression).
- Boot each per-depth .hda; screenshot at desktop; pixel-diff vs the MAME
  oracle snapshot of the same disk (the `docs/mame_compare.md` flow).
- 6.0.8 SCSI boot-to-Finder frame count vs the F2400 baseline (quantifies the
  CPU cost of slot stealing; expect ~0 at 1/2bpp, bounded slowdown at 8bpp).
- Floppy read during active video (dio slot stealing sanity).
- Then Quartus build → HW: same depth sweep, confirm color + no corruption +
  no white bar. (SDRAM latency is the HW-only variable; the FIFO's fill-ahead
  margin is the mitigation — fill target ≥ a few words, not just-in-time.)

## Phase 2 — true 60 Hz 640x480 (the "640x480 issue")
*(FPGA-centric; sim sees only parameter changes)*

- Add the spare PLL tap (outclk3, currently 0 MHz) at **25.175 MHz**.
- VGA mode: `CLK_VIDEO = clk_video25`, `CE_PIXEL = 1`, timing 800x525 → true
  59.94 Hz. The Phase-1 line FIFO becomes the clock-domain crossing (write
  port clk_sys / read port clk_video — make it a true dual-clock BRAM in
  Phase 1 so Phase 2 is just re-clocking the read side).
- 512x384 mode keeps the existing clk_sys/2 path (62.4 Hz ≈ the real 60.15 Hz
  — already scaler-friendly). CLK_VIDEO source muxed by monitor_id at reset/
  mode change only.
- Needs Quartus PLL regen on the Windows box + RT4K verification. Risk: ascal
  dislikes CLK_VIDEO switching — fallback is keeping 512x384 on the 25.175
  domain with an integer /2 CE (12.59 MHz → 48 Hz: NOT acceptable, so the
  fallback is actually shipping VGA-only on the new clock first).

## Phase 3 (deferred) — SDRAM burst reads for video
- `sdram.v` BURST_LENGTH 1→2/4 for video-slot reads only (8-state machine
  needs a widened data window — careful redesign at 65 MHz).
- Buys: 16bpp@512x384, and relieves the 8bpp CPU starvation from Phase 1.

## Color depth ceiling (HW-confirmed + VRAM math — 2026-06-05)

The V8 ASIC has EXACTLY 5 depth modes: 1/2/4/8/16 bpp (MAME v8.cpp modes 0-4).
There is **no 24bpp ("millions") mode** — never an LC capability. Confirmed on
the user's real LC: 2/4-color modes are **grays only**; real color starts at
**16 colors (4bpp)**. So "no color" is NOT a color-path bug — it is a direct
consequence of >2bpp being unusable today. Fix the fetch (Phase 1) → color.

512KB VRAM capacity (our config):

| depth            | 640x480 (default monitor) | 512x384 (12" RGB) |
|------------------|---------------------------|-------------------|
| 16-color  (4bpp) | 150KB ✓                   | 96KB ✓            |
| 256-color (8bpp) | 300KB ✓ — **640x480 max** | 192KB ✓           |
| thousands (16bpp)| 600KB ✗ won't fit         | 384KB ✓ — **512x384 max** |
| millions  (24bpp)| ✗ not a V8 mode           | ✗ not a V8 mode   |

- **16 + 256 colors at 640x480 = one piece of work (Phase 1).** 4bpp ≈40% bus,
  8bpp ≈80% bus (CPU slows during active video, like the real LC). No separate
  "add 256" step — both light up once the FIFO/arbiter lands.
- **Thousands (16bpp) = 512x384 ONLY, and needs Phase 3 (SDRAM burst):** even
  taking all 4 bus slots, single-word fetch tops out ~320 words/line; 16bpp at
  512 active px needs 512 words/line → must read ≥2 words/access. Deferred.
- **Millions (24bpp): OFF the roadmap** — not a V8 mode, VRAM can't hold it.

## Answers to the open questions
- **"Will this fix the 640x480 issue?"** Phase 1 fixes the corruption at any
  depth on that monitor; the *refresh-rate* part (38.7 Hz shimmer/scaler
  trouble) is Phase 2.
- **"Will this fix the white bar?"** Yes — Phase 1 item 3 removes the whole
  pre-first-word artifact class, palette-independent.
- **"No color"**: expected to be the combination of (1) >2bpp being unusable
  so color depths were never stable, and (4) the readback divergence possibly
  restricting what Monitors offers. Both addressed in Phase 1.

## Order of work
1. Phase 0 oracle prep (MAME per-depth disks + snapshots) — half a day.
2. Phase 1 in sim, validated against oracles, then Quartus build + HW sweep.
3. Phase 2 after Phase 1 is confirmed on HW (separate Quartus/PLL session).
4. Phase 3 only if 16bpp / 8bpp-speed demand it.
