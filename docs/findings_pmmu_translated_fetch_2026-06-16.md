# Findings + FIX: LC II PMMU bug #3 — spurious bus error on the post-pmove(TC) fetch

**Date:** 2026-06-16
**Branch:** `030_LCii`
**Status:** ROOT-CAUSED and **FIXED** in `rtl/tg68k/TG68KdotC_Kernel.vhd` (commit
`0590605`). Synced byte-identical to `../MacIIvi_MiSTer` (`e345d01`).
**Supersedes** the "first MMU-translated fetch faults / page tables are valid"
framing in `handoff_lcii_egret_pmmu_2026-06-16.md` and the bus-FSM/walker guesses.

## Symptom

After the boot enables caches + MMU and jumps to a ROM alias:

```
a41682: movec  D5,CACR          ; enable I/D caches
a416b2: pmove  (A3),tc          ; enable MMU (TC.E=1)
a416b6: jmp    (A5=$40A0010E)    ; jump to page-mapped ROM alias
```

…the CPU bus-errors, and with `VBR=0` the vector-2 fetch goes through zeroed RAM
at `$08` → PC=0 → garbage → `$00007FF8` wedge.

## Root cause (measured with `+define+PMMU_TRACE`, taps added to `tg68k.v`)

The faulting access is the **instruction fetch of the `jmp` itself at `$00a416b6`**
(fc=110, supervisor program) — *not* the jmp target. Captured at the trap:

```
PMMU REQ #0 addr=003fee00 va=00a416b6 fc=110   → 0009fc0a   (root table desc HIGH)
PMMU REQ #1 addr=003fee04 va=00a416b6 fc=110   → 003fedd0   (root table desc LOW = next table)
PMMU trap_berr addr=00a416b8 va=00a416b6 fstat=4401 tt0=0 tt1=0
```

`fstat=4401` = **L (limit) + I (invalid), level 1**. The root table descriptor
`0009fc0a` has L/U=0 (upper) and LIMIT = bits[30:16] = **9**, so the level-1 table
maps indices 0–9 only (identity `$000xxxxx–$009FFFFF`, ~10 MB). VA `$00a416b6` has
VA[31:20]=`0x00a`=**10** > 9 → limit violation. So **physical ROM `$00Axxxxx` is
deliberately unmapped**; TT0/TT1 are 0 (disabled). The `$40xxxxxx` alias (the jmp
target) *is* mapped via a different root entry (`003fee40`, confirmed by a later
walk `va=40a0010e`).

On real HW the residual fetch of `$00a416b6` is served from the **logical-tagged
68030 instruction cache / prefetch queue** (filled while the MMU was off, and the
I-cache was just enabled by `movec CACR`), so it bypasses translation. The canonical
`pmove TC; jmp <alias>` idiom relies on exactly this: the `jmp` is prefetched and
executes, and only its *target* is translated. TG68K is **uncached and refetches**
the next opcode after the pmove, so it re-translated the still-physical ROM PC and
limit-faulted.

(The earlier handoff's "the same VA re-walks and resolves" was a mis-read: the
re-walk was a *different* VA — `$001ff3c4`, fc=5 data — generated by the derail,
not `$00a416b6`.)

## FIX

Model the prefetch in the kernel (`TG68KdotC_Kernel.vhd`): when the MMU enable goes
live, **instruction fetches from the page the CPU was executing in bypass
translation (identity)**; the first instruction fetch that *leaves* that page (the
jmp target) re-arms translation and is walked normally.

- Arm a `mmu_fetch_grace` flag + latch `mmu_grace_page = TG68_PC[31:20]` on the
  rising edge of `pmmu_tc_en`; disarm on the first `state="00"` fetch whose page
  differs.
- `mmu_grace_suppress` gates BOTH `pmmu_req` (no walk/fault) and `addr_out` (mux to
  the identity logical address) for graced fetches. A **rising-edge term** covers
  the enable cycle itself — the registered grace flag only lands one `clkena` later,
  so without it the first prefetched fetch races ahead and still faults (this was
  the bug in the first fix attempt; `grace=800a` showed armed-but-too-late).
- Only instruction fetches (`state="00"`) are graced; data accesses always translate.

030-only (the LC's 68020 core has no PMMU, so it can't regress the LC).

## Verification

`releases/boot0-nomemcheck.rom`, `+define+PMMU_TRACE`, run to F110:
- `trap_berr @ $a416` : **0** (was the derail).
- The jmp target `$40A0010E` now walks via its own root entry and resolves.
- No `$00007FF8` wedge; execution proceeds into translated low-RAM and VASP I/O
  (`kpc=$50f14000`…).
- CPU single-step corpus (MacIIvi `SingleStepTests/tg68k`, golden corpus):
  **194/525/2 unchanged**, same 5 PRM-undefined-CCR diffs — no ISA regression.

Any remaining boot failure is a **new, downstream** blocker, not bug #3.

