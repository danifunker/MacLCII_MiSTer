# Plan: Move Mac LC VRAM into on-chip BRAM (kill the video bandwidth wall)

> Status: APPROVED 2026-06-07, branch `video-new-technique`. Written so a cold session
> can execute it end-to-end.

---

## 1. In plain English (read this first)

**The problem:** Color video is broken. At 16-color (4bpp) and above, only the left
~40% of the screen draws; the rest is garbage/white. Two separate teams chased this for
days down two different roads (shared SDRAM, then HPS DDR3) and **both dead-ended at the
exact same wall** for different reasons.

**The real cause:** The picture (the framebuffer) lives in *external* memory, so the
video hardware has to re-read the whole screen from far-away RAM every single frame,
fast enough to keep up with the beam. Both external memories are too slow/too shared to
do that. It was never about *total* bandwidth (the screen only needs ~9 MB/s) — it's
*latency and sharing*.

**The fix we're proving:** Stop using external memory for the picture. Put the
framebuffer **inside the FPGA**, in on-chip block RAM (BRAM). On-chip BRAM is
*dual-ported*: the CPU writes one door while the video reads the other door, **at the
same time, every clock, with zero waiting**. The whole "not enough bandwidth" problem
simply ceases to exist.

**Why we now know it fits:** The last two FPGA builds use only **73 of 553** on-chip RAM
blocks (13%). The HDMI scaler eats 43 of those; the actual Mac core uses just 15. That
leaves **~480 free blocks (~600 KB)**. Every video mode the Mac LC actually supports
fits in that space — including the hardest one we care about, **16bpp @ 512×384 (384
KB)**. (640×480 @ 16bpp is *not* a real Mac LC mode and is explicitly out of scope.)

**What success looks like:** Full-screen, correct-color desktop at 4bpp, 8bpp, and
16bpp@512×384 — and as a bonus the CPU gets *faster* because video stops stealing SDRAM
cycles. When we're done, the DDR3 experiment is retired and **no debug/JTAG probes
remain in the design.**

**How we'll prove it as we go:** Each build adds a temporary JTAG probe (SignalTap /
In-System Sources & Probes) to confirm one specific thing on real hardware — "CPU writes
are landing in BRAM," then "video now fetches the whole line," then "no read/write
corruption." Each probe is **removed before the next step**, and the final build is
verified probe-free.

## 3. Full diagnosis of the previous problems

### 3a. Shared 16-bit SDRAM path (the original)
Video time-shares one 16-bit SDRAM bus with the CPU on a 4-slot rotation, getting ~2 of
4 slots. Measured result: only **~64 of 160 words/line (~40%)** reach the screen at
4bpp. The `fix-pram` "video-priority" arbiter (video takes both video+extra slots, disk
yields, slot-0 reads through blanking, registered `video_req`) **did not move the wall** —
the per-slot delivery is the limit, not the slot *count*.

### 3b. HPS DDR3 path (`fix-pram`, the second attempt)
Ported X68000's `ddram.sv`, then a whole-line burst prefetch (`ddr_vram.sv`). The RTL was
*proven correct* (a position-gradient diagnostic with the burst-wait skipped rendered
full-width), yet 4bpp still stopped at **~64 words/line**. Root-caused on hardware:
- The HPS f2sdram `ram1` read **does not stream**: ~**276-cycle fixed latency + ~31
  cycles per 64-bit beat** ⇒ ~**6.8 MB/s** effective (~1% of DDR3).
- Doubling the controller clock to `clk_mem` (65 MHz) moved the boundary **zero pixels**
  ⇒ the cost is **wall-clock-bound**, not bridge cycles.
- MPFE arbitration/QoS **ruled out** by live `devmem` A/B of `mppriority`
  (0x0/0x9/0x3F/0xFFF — all identical).
- Structural: HPS FPGA-fabric ports are built for *cached/occasional* access (X68000 uses
  `ram1` for CPU-RAM caching). The only port designed to *stream* a framebuffer is the
  128-bit `vbuf` port — **owned by the HDMI scaler (ascal)**.

### 3c. The unifying insight
Both roads converge on the same ~64-word wall, by different mechanisms. The bandwidth
*required* is trivial (4bpp@640 = 160 words / 1600-cyc line ≈ **9 MB/s**). The defeats
were **latency + bus-sharing + read-serialization** — all artifacts of keeping the
framebuffer in external memory. Per-mode ceilings on external memory: 4bpp marginal,
8bpp needs two ports, 16bpp needs the scaler's wide port. That is a losing fight.

## 4. Hypothesis we are trying to prove

> **If the framebuffer lives in on-chip dual-port BRAM (CPU on port A, video on port B,
> single `clk_sys` domain), then video can read a pixel word every clock with no
> arbitration or latency, eliminating the bandwidth wall for every Mac LC video mode —
> and CPU throughput rises because video no longer competes for the SDRAM bus.**

Falsifiable predictions, checked on HW with probes (Section 6):
1. CPU framebuffer writes land in BRAM (write counter advances; readback matches).
2. Video fetch reaches **`words_per_line` every line** (no ~64 ceiling) at 4/8/16bpp.
3. Rendered screen is full-width and correct-color at all supported depths.
4. No read/write corruption (single-clock TDP is coherent by construction).

If (2) holds but the picture is wrong → it's a content/addressing bug, not bandwidth
(separable). If the design won't fit/route → fall back per Section 8.

---

## 5. Evidence it fits (from the two supplied fit reports)

Device **5CSEBA6U23I7**: **553 M10K blocks / 5,662,720 bits**. Both `alpha_1`
(pre-refactor) and `last_build` (DDR3 WIP) report **identical 73/553 (13%)** — the DDR3
detour cost ~0 BRAM. Breakdown: `ascal` scaler **43**, `emu` core **15** (dataController
12 ≈ Egret ROM/PRAM/ASC FIFO, Ariel CLUT 1), rest framework ~15. **→ ~480 free blocks
(~600 KB).**

1 M10K = 10,240 bits (1.25 KB). Framebuffer cost (ideal ↔ conservative ×16 packing),
supported modes only:

| Mode | Framebuffer | Blocks (ideal–×16) | + current 73 | /553 |
|---|---|---|---|---|
| 4bpp @ 640×480 | 150 KB | 120–150 | 193–223 | 35–40% |
| 8bpp @ 512×384 | 192 KB | 154–192 | 227–265 | 41–48% |
| 8bpp @ 640×480 (packed) | 300 KB | 240–300 | 313–373 | 57–67% |
| **16bpp @ 512×384** | **384 KB** | **307–384** | **380–457** | **69–83%** |

**Conclusion:** sizing the BRAM to **384 KB (16bpp@512×384)** covers *every* supported
mode and lands at 69–83% total M10K — fits with margin. ALMs are 60% used (40% free).

## 6. Implementation phases (each with a JTAG probe checkpoint)

> **JTAG mechanics for MiSTer:** probes attach over the DE10-Nano USB-Blaster JTAG.
> For a probe build, program the **`.sof`** with Quartus Programmer over JTAG (not the
> runtime `.rbf`), then use **Tools ▸ Signal Tap Logic Analyzer** (waveform capture) or
> **Tools ▸ In-System Sources and Probes Editor** (live read/write — this repo already
> uses ISSP, see the `MacLC.qsf` header comment). Probe-free verification uses the normal
> `.rbf` deploy (`scripts/build.sh` + `scripts/deploy_cmd.sh`).
> **Probe discipline:** every phase *adds* a probe, *verifies*, then *removes* it before
> the next phase. SignalTap: `set_global_assignment -name ENABLE_SIGNALTAP ON` +
> `-name USE_SIGNALTAP_FILE <f>.stp`. ISSP: a temporary `altsource_probe` instance in
> RTL. Removal = delete those assignments/instances + the `.stp`, then rebuild.

### Phase 0 — Prep (no probe)
- Cherry-pick the TG68 `MacLC.sdc` multicycle (Section 2). Confirm a clean baseline
  build: all STA slacks positive; `verilator` boots (frame-350 grey/black memory-test
  pattern per `CLAUDE.md`).
- Shelve the DDR3 path: ensure no `ddr_vram`/`ddram` is in this branch's `files.qip`.

### Phase 1 — Dual-port BRAM VRAM module + CPU write/read path
- **New `rtl/vram_bram.sv`:** single-clock (`clk_sys`) true-dual-port M10K,
  `(* ramstyle="M10K" *)`, 16-bit words + byte enables, depth = 384 KB / 2 =
  **196,608 words**. Port A = CPU (read+write, byte-masked). Port B = video read.
- **Route CPU VRAM (`selectVRAM`) to port A** instead of SDRAM: writes use
  `{~_cpuUDS,~_cpuLDS}` byte enables + `memoryDataOut`; reads return BRAM data into the
  CPU data path (`dataController_top` `memoryDataIn` mux — add a VRAM-from-BRAM source).
  BRAM read is 1-cycle synchronous (faster than the SDRAM path it replaces), so DTACK
  timing only relaxes; verify in sim.
- **Addressing / stride:** 512×384 maps 1:1 (offset `cpuAddr[19:0]-$40000 → word`).
  For the **640-wide** modes (1024-byte stride, but FB sized to 384 KB) apply active-only
  **packing**: `line = off>>10; col = off & $3FF; packed_word = (line*active_words)+col`
  (`active_words` per depth; a constant-ish multiply → DSP, 44/112 used). Keep the
  translate in one place in `addrController_top.v`.
- **Both tops:** instantiate `vram_bram` in `MacLC.sv` *and* `verilator/sim.v`; remove
  VRAM from the SDRAM region in both (and from `sim_ram.v` usage). Sim models BRAM
  natively (reg array).
- **🔌 Probe 1 (ISSP):** temporary `altsource_probe` exposing a 32-bit `vram_wr_count`
  and `last_wr_addr`/`last_wr_data`, plus a *source* to drive a read address and a
  *probe* to read back that BRAM word. **Verify on HW:** count climbs during boot;
  readback of a known framebuffer word matches what the CPU wrote.
- **🧹 Remove Probe 1** (delete the `altsource_probe` instance); rebuild before Phase 2.
- **Sim gate:** frame-350 pattern still correct; `check_boot.sh --run 100` PASS.

### Phase 2 — Video fetch from BRAM port B (the actual fix)
- **`rtl/maclc_v8_video.sv`:** change the line-buffer fill *source* from the SDRAM slot
  to **`vram_bram` port B** — read one word per `clk_sys` with **no slot gating**. The
  whole line (≤640 words) now fills in ≤640 of the ~1600 line-clocks → always completes.
  Keep the existing ping-pong `linebuf` + display path unchanged (lowest-risk).
- **`rtl/addrController_top.v`:** remove video from the SDRAM arbiter — drop the
  `videoBusControl` read, `video_extra`, `v8_video_fetch`, `vid_sdram_word`, and the
  `_ramOE` video terms. Freed slots fall back to CPU/disk automatically.
- **Both tops** updated; re-audit `docs/verilator_differences.md`.
- **🔌 Probe 2 (SignalTap):** capture `fetch_idx` (max per line), `words_per_line`, and a
  "line complete" flag; trigger at end-of-line. **Verify on HW:** `fetch_idx` reaches
  `words_per_line` every line at 4bpp **and** 8bpp **and** 16bpp@512 — the ~64 ceiling is
  gone. (Complementary visual: the position-gradient render trick should now fill full
  width — not a JTAG probe, document it but don't ship it.)
- **🧹 Remove Probe 2** (`ENABLE_SIGNALTAP OFF`, drop `USE_SIGNALTAP_FILE`, delete
  `.stp`); rebuild.
- **Sim gate:** forced 4/8/16bpp test pattern renders **full-width**; boot pattern intact.

### Phase 3 — Real color + coherency
- Wire real BRAM data through to pixels (it already is, via the line buffer). Boot to the
  Finder at each depth; compare framebuffer to a **MAME `maclc` oracle** at matching
  depth (`docs/mame_compare.md`).
- **🔌 Probe 3 (SignalTap):** trigger on simultaneous port-A write + port-B read to the
  *same* word address; capture both to confirm no torn/stale read (expected: none —
  single-clock TDP is coherent). Lightweight; one capture session.
- **🧹 Remove Probe 3**; rebuild.
- **HW:** screenshots at 4bpp@640, 8bpp@512, 8bpp@640, **16bpp@512×384** — all
  full-width, correct color.

### Phase 4 — Probe hygiene + final clean build (no probes anywhere)
- Sweep for **zero** debug logic:
  - `MacLC.qsf`: no `ENABLE_SIGNALTAP`, no `USE_SIGNALTAP_FILE`, no SignalTap/ISSP debug
    assignments; delete any `*.stp`.
  - RTL: `grep -rIE 'altsource_probe|sld_|signaltap|stp_|debug_probe' rtl/ MacLC.sv` → no
    matches.
  - Fit report of the final `.rbf` build: **no** `sld_signaltap`, `altsource_probe`,
    `sld_virtual_jtag`, or SLD hub instances; M10K count = baseline 73 **+ framebuffer
    only** (≈ 380–457), no stray debug blocks.
- Final `.rbf` build: all STA slacks positive (setup + hold); deploy; re-screenshot all
  depths to confirm the probe-free image matches the probe builds.

## 8. Risks & fallbacks
- **CPU VRAM read timing** moving SDRAM→BRAM: BRAM is faster, so DTACK only relaxes;
  validate the read-data mux + DTACK in sim before HW. *Risk: low.*
- **640-wide packing multiply**: if the constant-mult adds timing pressure, organize the
  framebuffer as wider words (read video 32/64-bit) — also improves M10K packing toward
  the ideal (~69%). *Fallback available.*
- **16bpp@512 at ~83% M10K (×16 worst case):** if routing is hard, switch to wider memory
  words (→ ~69%) or ship 8bpp@512 (48%, rock-solid) and treat 16bpp as a follow-up.
- **Both-tops drift:** every signal change goes into `MacLC.sv` *and* `verilator/sim.v`;
  update `docs/verilator_differences.md`. *Hard requirement.*
- **Probe leakage:** Phase 4 grep + fit-report check is the gate; do not tag a release
  until it passes.
